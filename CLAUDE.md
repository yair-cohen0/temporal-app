# CLAUDE.md ā€” Temporal Workflow Engine

This file is for LLM agents working in this repo. It documents constraints, non-obvious decisions, and invariants that are not derivable from reading the code alone.

## What this service is

A generic Temporal-based workflow engine. It exposes a REST API for starting and managing Temporal workflow executions, and runs a Worker that executes workflow + activity code. This is one microservice in a larger system ā€” do not assume knowledge of any consumer service, do not call external APIs from this codebase, and do not reference any specific consumer service by name.

## Hard Constraint: Worker writes to outbox only

**The Worker writes to the `outbox` MongoDB collection only. It never reads from Mongo, never accesses any other collection.**

This is enforced at the type level: `src/shared/mongo.ts` exports exactly one function ā€” `getOutboxCollection()`. It does not export the database object, the client, or any other collection. Any future need for broader access must be a deliberate, explicit decision.

Why: the outbox is the sole side-effect channel. Everything the Worker does that affects the outside world must flow through the outbox so the consuming service has a complete, auditable record of intended actions.

## ID Conventions

| ID | Type | Identifies | Generated by | When |
|---|---|---|---|---|
| `_id` | ObjectId | The Mongo document itself | Auto by Mongo on every insert | At insert time |
| `workflowId` | string (UUID v4) | A workflow execution from the user's perspective ā€” one logical instance, even across resets | Caller (out of scope of this service) | Before calling the start route |
| `runId` | string | One Temporal-internal attempt at running the workflow. Changes on reset / continue-as-new | Temporal | Returned from start call; available in activities via `Context.current().info.workflowExecution.runId` |
| `actionId` | string | One specific action within one run. For step-related actions, equals `stepId` (or `${stepId}:timeout` for timeouts). The unique index on this field is what enforces activity idempotency. | Caller for `createFlow` / `forwardSignal`; workflow author (via `stepId`) for step-related actions | At creation of the corresponding outbox doc |
| `stepId` | string | A specific step instance in a workflow. Semantic, human-readable, e.g. `managerApproval`, `legalReview-2` | Workflow author when invoking a step primitive | When the step is invoked |
| `actorId` | string | The user who performed an action | Caller (from auth session) | When user clicks approve/sign/etc. |

`actionId` and `stepId` are conceptually distinct even though they are equal for step-related actions. Do not merge them into one field.

## Idempotency in writeOutboxDocument

The `writeOutboxDocument` activity is called with Temporal's default retry policy (indefinite retries). A successful Mongo insert whose acknowledgement is lost to a network partition will be retried. Idempotency is enforced via:

1. A **unique index on `actionId`** in the outbox collection (created at startup by `getOutboxCollection()`).
2. On `MongoServerError` with code 11000 (duplicate key), the activity treats the write as success.

**Why `_id` is NOT the idempotency key:** Mongo auto-generates a new `ObjectId` on every `insertOne` call, so a retried insert would always get a new `_id` and bypass duplicate detection. `actionId` is stable across retries because it is derived from `stepId` (set by the workflow author, deterministic in workflow code).

For step primitives: `actionId === stepId` for the initial outbox doc; `actionId === "${stepId}:timeout"` for the timeout outbox doc. These two are distinct so they don't collide with each other.

## 409 on Duplicate workflowId

When starting a workflow, if a workflow with the same `workflowId` is already running, Temporal throws `WorkflowExecutionAlreadyStartedError`. The start route catches this and returns HTTP 409 with error code `WORKFLOW_ID_CONFLICT`.

Callers treat 409 as success ("already started"). This enables idempotent retries: if the API server crashes after Temporal persists but before returning the 201, the caller retries with the same `workflowId` and gets 409, which it handles identically to 201.

## Why workflowIds are always client-supplied

The API never generates `workflowId` server-side. The caller generates it before calling the start route. This is the only way to achieve idempotent start semantics ā€” the caller must know the ID in advance so it can retry and detect the "already exists" case.

## Mongo module ā€” only getOutboxCollection()

`src/shared/mongo.ts` exports exactly one function: `getOutboxCollection(): Promise<Collection<OutboxDoc>>`. It does NOT export:
- The `MongoClient` instance
- The `Db` object
- Any other collection accessor

This is intentional type-level enforcement. If another module imports from `mongo.ts`, TypeScript will only allow it to call `getOutboxCollection()`. There is no way to reach another collection or to read from Mongo via this module.

## Outbox document schema

```
OutboxDoc ā€” discriminated union keyed on actionType
Each variant pairs one literal actionType with its matching actionConfig.

actionType values:
  createFlow         ā€” workflow was started
  forwardSignal      ā€” a signal was forwarded
  awaitGroupApproval ā€” waiting for any user in a group
  awaitRankApproval  ā€” waiting for any user of a rank
  awaitSignature     ā€” waiting for a specific user
  endpoint           ā€” fire-and-forget resource grant
  timeout            ā€” a step timed out
```

The `OutboxDocInput` type uses a distributive Omit (not standard `Omit`) to preserve discriminant narrowing across union members. Standard `Omit<A|B, K>` would flatten the union and allow mismatched `actionType`/`actionConfig` pairs.

The Worker only ever creates docs with `status: 'pending'`. Status transitions to `processing`, `done`, `failed` are made by the consumer service (out of scope).

## Signal routing ā€” stepCompleted

The signal name is `stepCompleted`. All step types use this single signal. Signal payloads carry a `stepId` field that identifies which step instance the signal is for.

The signal handler in `src/shared/steps.ts` maintains a module-level `Map<stepId, SignalPayload>`. Each waiting step calls `condition(() => receivedSignals.has(its-stepId))`. This is safe because:
- Each workflow execution gets its own V8 isolate (module state is per-execution)
- On replay, Temporal re-dispatches signals from history, repopulating the map correctly

## How step primitives work internally

Each waiting step (groupApproverStep, rankApproverStep, signatureStep):
1. Reads `workflowId` and `runId` from `workflowInfo()`
2. Calls `writeOutboxDocument` activity with the appropriate `actionType` and `actionConfig`, using `stepId` as `actionId`
3. Calls `condition(() => receivedSignals.has(stepId), timeoutMs)` ā€” blocks until the signal arrives or timeout
4. On signal: returns the `SignalPayload`
5. On timeout: calls `writeOutboxDocument` again with `actionType: 'timeout'` and `actionId: "${stepId}:timeout"`, then throws `ApplicationFailure.nonRetryable(..., 'StepTimeoutError')`

Workflow authors catch timeouts with:
```ts
import { ApplicationFailure } from '@temporalio/workflow';
try {
  await groupApproverStep(...)
} catch (err) {
  if (err instanceof ApplicationFailure && err.type === 'StepTimeoutError') { ... }
}
```

`endpointStep` uses `uuid4()` from `@temporalio/workflow` (deterministic in workflow context) as its `actionId`.

## Activity retry behavior

Activity retry policy is left at Temporal's default: indefinite retries with exponential backoff starting at 1 second. This is why `writeOutboxDocument` MUST be idempotent (see above).

## steps.ts ā€” workflow sandbox constraints

`src/shared/steps.ts` runs inside Temporal's V8 workflow sandbox. Rules:
- Only imports from `@temporalio/workflow`
- All other imports use `import type` (erased at compile time, no runtime Node.js code loaded)
- No Node.js built-ins, no Date.now(), no Math.random(), no fs/http/net
- `proxyActivities()` is called at module scope (safe inside the sandbox)

## resetWorkflow ā€” no high-level API

The Temporal TypeScript SDK (as of v1.x) has no high-level `resetWorkflow` method on `WorkflowHandle`. The `/reset` route uses `client.workflowService.resetWorkflowExecution(...)` (raw gRPC). This is expected behavior, not a bug.

## any casts in routes/workflows.ts

Several `// eslint-disable-next-line @typescript-eslint/no-explicit-any` comments exist in the routes file for:
- gRPC service method calls ā€” the callback/promise overloads produce an unresolvable `void & Promise<T>` intersection type
